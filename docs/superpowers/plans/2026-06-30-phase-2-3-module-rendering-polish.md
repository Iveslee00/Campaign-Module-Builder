# Phase 2+3 Module Rendering And Visual Polish Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stabilize NEXORA Builder module rendering so Builder, Preview, and Export stay structurally aligned, then improve high-risk product module visual output.

**Architecture:** Phase 2 hardens the current registry/exporter architecture with stricter DOM parity checks before deeper shared-component migration. Phase 3 applies targeted visual and layout fixes only after tests describe the expected exported structure. The immediate strategy is to stop known export regressions first, then continue migrating modules toward a shared renderer contract.

**Tech Stack:** Next.js 14, React 18, TypeScript, static exporter functions, custom verification scripts, scoped CSS generated by `lib/export/cssGenerator.ts`.

---

## File Structure

- `lib/modules/moduleRegistry.ts` remains the export registry entry point.
- `modules/preview/*Preview.tsx` remains the Builder/Preview rendering layer for now.
- `modules/exporters/*Exporter.ts` remains the Export rendering layer for now, but must match preview DOM contracts.
- `lib/export/cssGenerator.ts` remains the scoped export CSS source.
- `scripts/verify-full-module-export-stability.mjs` becomes the broad structural guard.
- `scripts/verify-module-export-parity.mjs` is introduced as the all-module preview/export contract guard.
- `scripts/verify-product-export-parity.mjs` is introduced for module-specific preview/export parity checks.
- `docs/module-development-standard.md` documents the stricter rule: any module edit must update preview, exporter, CSS, and parity tests together.
- `docs/module-rendering-export-architecture.md` tracks current Phase 2 state and the future shared-component target.

## Post-Task Notes

This plan intentionally does not claim the full shared React component renderer is complete. That is the deeper Phase 2 migration after the current high-risk export bugs are blocked by tests. The immediate work prevents tomorrow's user-facing export defects while preserving a path to true shared rendering.
